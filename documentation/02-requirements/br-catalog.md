# Каталог бизнес‑требований (BR)

Ниже приведен развернутый иерархический список BR из `business-requirements.md` для ведения учёта и статусов.

### Критический приоритет (Must Have)
- BR-001: Создание интеллектуальной системы обучения
- BR-002: Обработка текстовых запросов пользователей
- BR-003: Проектно-ориентированная работа с данными

### Высокий приоритет (Should Have)
- BR-005: Извлечение данных из документов (OCR)
- BR-006: Формирование отчетов в Excel
- BR-007: Работа с нормативной базой (ГОСТы)

### Средний приоритет (Could Have)
- BR-008: Обработка специализированных форматов (DWG/сметы)

### Дополнительные бизнес‑требования из входных документов (1–8)
- BR-014: Извлечение реквизитов из главного штампа (PDF/DWG)
- BR-015: Табличное содержание спецификаций (PDF/DWG)
- BR-016: Нормализация смет (ARP/GSFX/XML/XLS)
- BR-017: Выжимки из нормативных документов (DOCX → PDF)
- BR-018: Отчёты по шаблонам подразделений (XLSX)

### Низкий приоритет (Won't Have в текущем этапе)
- BR-012: Пользовательские интерфейсы (out of scope на этом этапе) (будет внутренний API для доступа к данным из внешних систем)
- BR-013: Интеграция с существующими системами (out of scope на этом этапе)

### Обновленные BR, добавленные в результате интервью
- BR-019: Пользователь определяет область работы для системы (файл, каталог файлов, проект). Система предлагает список прикладных задач в соответствии с областью применения. Если прикладной задачи нет в списке, пользователю (учитель) предлагается обучить систему новой прикладной задаче. Новые прикладные задачи должны проходить путь валидации менеджментом организации, образно прикладная задача должна быть заапрувлена. 
- BR-020: Пользователь (учитель) определяет область работы для системы (файл, каталог файлов, проект). Перед/во время/или после обучения системы учитель добавляет определенные ГОСТы для этой проектной области. Учитель может обновлять ГОСТы если это требуется, система должна адоптировать прикладные задачи в соответствии с изменениями в ГОСТ



---

Примечание: процессные/управленческие пункты ведутся как `PT-###` в `work-plan.md` и не включаются в реестр BR.
