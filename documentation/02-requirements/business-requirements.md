## 1. Введение

### Цель документа
Данный документ определяет бизнес-требования для проекта AI GOST - интеллектуальной системы обучения на корпоративных документах. Система работает в проектно-ориентированном контексте и позволяет пользователям делать текстовые запросы для извлечения данных и поиска аналогов по проектам.

### Связь с другими документами
- **Vision Document** - общее видение и цели проекта
- **Договор №24/25** - формальные обязательства и ограничения
- **Use Cases** - детальные сценарии использования
 - **Глоссарий терминов** — см. раздел «1.3 Термины и сокращения» в SRS (`technical-requirements.md`)

### Термины и сокращения (кратко)
Ключевые термины, используемые в этом документе. Полный глоссарий — в SRS 1.3.
- OCR — распознавание текста из изображений/сканов/PDF
- CER/WER — доля ошибок распознавания по символам/словам
- NLP/классификация — отнесение документов к тематикам
- macro‑F1 — усреднённая метрика качества классификации по темам
- Структурирование — приведение данных к JSON/CSV/XLSX
- Валидность — доля записей, прошедших JSON Schema без ошибок
- Время обработки — секунды на документ (end‑to‑end)
- DWG/DXF — форматы чертежей; ARP/GSFX — сметные форматы
- JSON Schema — схема валидации структуры/типов
- ACL — разграничение доступа (списки управления доступом)
- **Учитель системы** — роль для загрузки проверенных документов и обучения системы
- **Пользователь системы** — роль для выполнения текстовых запросов и получения данных
- **Проектно-ориентированная работа** — привязка всех данных к конкретным проектам
- **Текстовые запросы** — естественные запросы пользователей на русском языке
- **Поиск аналогов** — поиск похожих проектов и извлечение данных по ним

## 2. Бизнес-требования по приоритетам

### КРИТИЧЕСКИЙ ПРИОРИТЕТ (Must Have)

#### BR-001: Создание интеллектуальной системы обучения
**Описание**: Разработать систему, которая "учится" на корпоративных документах и накапливает знания  
**Обоснование**: Основа для всех последующих функций системы  
**Критерии приемки**:
- Система может изучать загруженные документы
- Система накапливает знания о проектах и технических решениях
- Система может отвечать на текстовые запросы пользователей
- Система работает в проектно-ориентированном контексте

#### BR-002: Обработка текстовых запросов пользователей
**Описание**: Обеспечить возможность делать естественные текстовые запросы на русском языке  
**Обоснование**: Основной интерфейс взаимодействия с системой  
**Критерии приемки**:
- Пользователи могут делать запросы типа "вытащи данные про козырек"
- Система понимает контекст проектов в запросах
- Система формирует структурированные ответы в Excel
- Время обработки запроса не превышает 5 секунд

#### BR-003: Проектно-ориентированная работа с данными
**Описание**: Обеспечить привязку всех данных к конкретным проектам  
**Обоснование**: Критично для поиска аналогов и извлечения данных по проектам  
**Критерии приемки**:
- Все документы привязаны к конкретным проектам (подстанция, козырек и т.д.)
- Система может извлекать ведомости объемов работ по проекту
- Система может формировать комплекты рабочей документации по проекту
- Система поддерживает поиск аналогов по проектам

#### BR-004: Создание технического задания
**Описание**: Подготовить полное техническое задание для разработки производственной системы  
**Обоснование**: Контрактное обязательство и основа для следующего этапа  
**Критерии приемки**:
- ТЗ содержит все функциональные требования
- ТЗ содержит все нефункциональные требования  
- ТЗ включает архитектурную схему решения
- ТЗ согласовано и принято заказчиком

### ВЫСОКИЙ ПРИОРИТЕТ (Should Have)

#### BR-005: Извлечение данных из документов (OCR)
**Описание**: Обеспечить автоматическое извлечение текста и данных из документов  
**Обоснование**: Необходимо для обучения системы и обработки запросов  
**Критерии приемки**:
- Создан прототип OCR для PDF и изображений
- Протестирована точность распознавания на образцах заказчика
- Определены ограничения и граничные случаи
- Задокументированы рекомендации по выбору OCR-технологий

#### BR-006: Формирование отчетов в Excel
**Описание**: Обеспечить создание структурированных отчетов в Excel по запросам пользователей  
**Обоснование**: Основной выходной формат для структурированных данных  
**Критерии приемки**:
- Система формирует отчеты в Excel по текстовым запросам
- Отчеты содержат корректно структурированные данные
- Поддерживается форматирование таблиц и диаграмм
- Время формирования отчета не превышает 10 секунд

#### BR-007: Работа с нормативной базой (ГОСТы)
**Описание**: Обеспечить изучение и использование нормативных документов для валидации  
**Обоснование**: Критично для корректной работы с техническими документами  
**Критерии приемки**:
- Система изучает ГОСТы и нормативные документы
- Система использует нормативную базу для валидации данных
- Поддерживается поиск по нормативным требованиям
- Задокументированы процедуры обновления нормативной базы

#### BR-008: Архитектурное проектирование
**Описание**: Разработать архитектуру будущей производственной системы  
**Обоснование**: Необходимо для планирования разработки и интеграции  
**Критерии приемки**:
- Создана высокоуровневая архитектурная схема
- Определены компоненты системы и их взаимодействие
- Спроектированы интерфейсы интеграции
- Задокументированы технические решения

### СРЕДНИЙ ПРИОРИТЕТ (Could Have)

- Проанализированы решения для обучения на документах

## 2.2 Дополнительные бизнес‑требования из входных документов (1–8)

#### BR-014: Извлечение реквизитов из главного штампа (PDF/DWG)
**Описание**: Система извлекает реквизиты из главного штампа (title, cipher, sheetNumber, documentName, sheetFormat, designStage, organization, documentDate, revisionNumber) для PDF (МЧ/СО) и DWG.  
**Роли/подразделения**: Инженер, ПТО  
**Бизнес‑цели**: Формирование сводной спецификации; унификация спецификаций; подготовка к бюджетированию.  
**Выход**: XLS по шаблону ПТО.

#### BR-015: Табличное содержание спецификаций (PDF/DWG)
**Описание**: Автоматическое извлечение табличного содержания спецификаций, включая расчетные формулы (где применимо).  
**Роли/подразделения**: Инженер, ПТО  
**Бизнес‑цели**: Сводные спецификации по проектам; унификация.  
**Выход**: XLS.

#### BR-016: Нормализация смет (ARP/GSFX/XML/XLS)
**Описание**: Парсинг сметных документов в единую модель Estimate с привязкой к шифру РД и проверкой totals/единиц/обязательных полей; сохранение формул, если доступны.  
**Роли/подразделения**: Инженер‑сметчик, Сметный отдел  
**Бизнес‑цели**: Анализ и корректировка смет; формирование бюджета.  
**Выход**: Нормализованный JSON + XLS.

#### BR-017: Выжимки из нормативных документов (DOCX → PDF)
**Описание**: Извлечение реквизитов нормативных документов (наименование, номер, дата утверждения) и формирование выжимок по запросу; экспорт в PDF.  
**Роли/подразделения**: Юрист, Юр. отдел  
**Бизнес‑цели**: Работа с нормативными документами; подготовка сопутствующих документов.  
**Выход**: PDF + ссылки на источники.

#### BR-018: Отчёты по шаблонам подразделений (XLSX)
**Описание**: Генерация Excel‑отчетов по утвержденным шаблонам ПТО и Сметного отдела, с проверкой структуры и форматирования.  
**Роли/подразделения**: ПТО, Сметный отдел  
**Бизнес‑цели**: Сводки, анализ, бюджетирование.  
**Выход**: XLSX (template‑compliant).

**Описание**: Определить требования по информационной безопасности и соответствию нормативам  
**Обоснование**: Критично для работы с конфиденциальными документами  
**Критерии приемки**:
- Проанализированы требования по защите данных
- Определены требования соответствия отраслевым стандартам
- Подготовлены рекомендации по обеспечению безопасности
- Задокументированы процедуры аудита и контроля

## Data Sources and Formats (consolidated from prototype inputs 1–8)

- PDF (machine-readable) — title block fields and tabular content extraction.
- PDF (scanned image) — same as above, but with OCR pre-processing (deskew, denoise, binarization).
- DWG (AutoCAD) — title block fields and table-like content extraction from text layers/blocks.
- Estimates: ARP / GSFX / XML / XLS — parsing to a unified Estimate model with formulas/totals validation; binding to RD cipher.
- DOCX (normative) — metadata extraction (name, number, approval date) and query-based excerpts; export to PDF.
- XLSX (target) — reference templates for PTO and Cost Estimation departments; report generation with formatting validation.

### Header Stamp Fields (for PDF/DWG)
- title, cipher, sheetNumber, documentName, sheetFormat, designStage, organization, documentDate, revisionNumber.

## 3. Нефункциональные требования

### Производительность
- **NFR-001**: Прототипы должны обрабатывать тестовые документы за разумное время (не более 30 сек на документ)
- **NFR-002**: Система должна поддерживать пакетную обработку документов

### Надежность  
- **NFR-003**: Прототипы должны стабильно работать на предоставленных образцах данных
- **NFR-004**: Система должна корректно обрабатывать ошибки и исключения

### Безопасность
- **NFR-005**: Все данные заказчика должны обрабатываться с соблюдением конфиденциальности на уровне коммерческих документов
- **NFR-006**: Доступ к прототипам должен быть ограничен участниками проекта
- **NFR-007**: Система должна работать локально без доступа к интернету

### Совместимость
- **NFR-008**: Прототипы должны работать с основными форматами документов (PDF, JPG, PNG, DWG)
- **NFR-009**: Архитектурные решения должны учитывать возможность интеграции с существующими системами

## 4. Ограничения и допущения

### Ограничения проекта
- **Временные**: 30 календарных дней на выполнение всех работ
- **Бюджетные**: 615 000 рублей (фиксированная стоимость)
- **Ресурсные**: Команда из 4 специалистов (132 часа общих трудозатрат)
- **Технические**: Ограничения существующей ИТ-инфраструктуры заказчика

### Допущения
- Заказчик предоставит качественные образцы документов для обучения системы
- Ключевые стейкхолдеры будут доступны для интервью в первые 2 недели проекта  
- Существующая ИТ-инфраструктура заказчика позволяет интеграцию AI-решений
- Нет критических нормативных ограничений, препятствующих использованию AI-технологий
- **Обучение системы**: Заказчик предоставляет проверенные документы для обучения системы
- **Текстовые запросы**: Пользователи будут делать запросы на русском языке
- **Проектно-ориентированность**: Все данные можно привязать к конкретным проектам
- **Локальная работа**: Система будет работать без доступа к интернету

## 5. Критерии приемки проекта

### Обязательные результаты
1. **Техническое задание** - полное, детальное, согласованное с заказчиком
2. **Отчет по анализу AI-моделей** - с обоснованными рекомендациями
3. **Рабочие прототипы** - минимум 3 прототипа по ключевым направлениям
4. **Архитектурная документация** - схемы и описания технических решений

### Критерии качества
- Все документы соответствуют стандартам оформления технической документации
- Прототипы демонстрируют заявленную функциональность на тестовых данных
- Рекомендации обоснованы результатами тестирования и анализа
- Техническое задание содержит достаточно деталей для начала разработки
- **Интеллектуальная система**: Система корректно отвечает на текстовые запросы
- **Обучение системы**: Система успешно изучает новые документы
- **Проектно-ориентированность**: Данные корректно привязаны к проектам
- **Excel отчеты**: Формируются корректно структурированные отчеты

Подробные метрики и статусы согласования см. в документе [metrics.md](metrics.md).

### POC: критерии успеха и решения
- Успех POC — это подтвержденная/опровергнутая осуществимость (feasibility) и чёткие выводы по ограничениям и следующими шагами.
- По каждому направлению фиксируется решение: Proceed / Iterate / Pivot / Stop с обоснованием.
- Допускается недостижение целевых метрик, если получены воспроизводимые результаты, зафиксированы причины и рекомендации по улучшению (стоимость/сроки/риски).

### Процедура приемки
1. Внутренняя проверка всех результатов командой проекта
2. Презентация результатов заказчику
3. Период рассмотрения заказчиком (до 10 рабочих дней)
4. Устранение замечаний (при наличии)
5. Подписание акта приемки работ

## 6. Риски выполнения требований

### Высокие риски
**Недоступность стейкхолдеров** → Невозможность выполнить BR-001  
*Митигация*: Раннее планирование интервью, альтернативные форматы сбора требований

**Отсутствие образцов данных** → Невозможность обучить систему (BR-001, BR-002)  
*Митигация*: Запрос образцов на старте проекта, использование синтетических данных

**Сложность обучения системы** → Низкое качество ответов на текстовые запросы (BR-002)  
*Митигация*: Поэтапное обучение, валидация на каждом этапе

### Средние риски  
**Технические ограничения форматов** → Невозможность реализации BR-009 (DWG/сметы)  
*Митигация*: Раннее тестирование, подготовка альтернативных решений

**Расширение объема работ** → Превышение временных и бюджетных ограничений  
*Митигация*: Строгое управление изменениями, приоритизация требований

**Сложность проектно-ориентированной работы** → Невозможность привязать данные к проектам (BR-003)  
*Митигация*: Ранний анализ структуры проектов, упрощение модели данных
